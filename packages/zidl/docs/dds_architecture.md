# DDS Architecture and Scope

Design decisions, DDS-profile scope, interoperability strategy, and the
long-term role of zidl in a Zig-native DDS implementation.

---

## What zidl Is (and Isn't)

zidl is the **IDL compiler and code generator** for a forthcoming Zig-native DDS
stack. It is not itself a DDS implementation. Its artifacts feed into the DDS
runtime:

```
zidl (this repo)
  └── generates: type definitions + CDR serializers + TypeObjects
  └── generates: DDS binding layer (--generate-interfaces)

zidl-rt / zidl-cdr / zidl-xtypes (companion packages in this repo)
  └── runtime CDR encode/decode used by generated code

[future: Zig DDS runtime]
  └── RTPS participant, transport, discovery, QoS
  └── DataWriter / DataReader implementation
  └── TypeObject registration and type discovery
```

The DDS binding layer (`--generate-interfaces`) generates the abstract API
stubs (DataWriter, DataReader, TypeSupport, etc.) that the DDS runtime will
implement as concrete types. The generated stubs define the required interface
shape; the runtime provides the vtable implementations.

## Full DDS vs XRCE

### Full DDS Profile (`--profile full`, the default)

- XCDR2 encoding with @appendable and @mutable support.
- Unbounded sequences, strings, and maps.
- TypeObject/TypeIdentifier generation (MinimalTypeObject XCDR2 LE streams).
- Heap allocation in generated code.
- Full DCPS API (`--generate-interfaces`): Entity, DomainParticipant, Publisher,
  DataWriter, Subscriber, DataReader, Topic hierarchy.
- Targeting DDS v1.4 (formal/15-04-10) compliance.

### XRCE Profile (`--profile xrce`)

Subset for bare-metal microcontrollers (DDS-XRCE v1.0, formal/20-02-01):

- XCDR1 encoding only (no DHEADER, no EMHEADER).
- Only `@final` types allowed (no @appendable, no @mutable).
- Only bounded sequences (`sequence<T, N>`); no unbounded sequences.
- No `map<K,V>`.
- No TypeObject/TypeIdentifier.
- No heap allocation in generated code.

XRCE profile constraints are validated before backends run (`validateXrce` in
`src/backend/interface.zig`). Violation is a compile-time error, not a runtime
check.

Current implementation note: this profile is a validation/codegen profile, not a
complete MicroZig or freestanding output target. The Zig backend can emit
Zig 0.15.1-compatible output via `--zig-version 0.15.1`, and bounded
strings/sequences use fixed-capacity `zidl_rt.BoundedArray` storage. A fully
freestanding/no-heap runtime path for MicroZig clients remains planned work.

## Interoperability Testing

### Cyclone DDS Interop (`interop/`)

The `interop/` directory contains a byte-level interoperability harness that
validates CDR stream compatibility between zidl-generated code and Cyclone DDS 11.0.1.

**Setup:** Requires a Cyclone DDS checkout; path configured in `interop/Makefile`.

**Test strategy:**
1. Write IDL types that both zidl and Cyclone DDS understand.
2. Serialize sample values using zidl-generated C code.
3. Deserialize the same bytes using Cyclone DDS participant.
4. Compare field values; any mismatch is a failure.

**Validated:**
- Encap header byte values (CDR1_LE = 0x0001, CDR2_LE = 0x0007).
- Primitive type encoding (all IDL primitive types).
- String encoding (length-prefixed, NUL-terminated).
- Struct member layout and padding.
- @appendable DHEADER framing.
- TypeObject EquivalenceHash byte values (confirmed matching Cyclone's computed hash).

**Running:**
```sh
make -C interop test
```

**Tests:** 10.

### zidl-rt / zidl-cdr Cross-Validation

`packages/zidl-cdr/` has 30 tests that cross-validate the C99 CDR library
against `zidl-rt` (Zig) byte-for-byte. These run as part of the normal
`zig build test` suite.

## TypeObject and Type Discovery

TypeObjects generated by zidl are intended to be registered with a DDS domain
participant and used for type discovery in the DDS-XTypes protocol.

**EquivalenceHash** (`equivalence_hash: [14]u8`): This is the on-wire
identifier used by Cyclone DDS, FastDDS, and other DDS implementations to
match types. zidl-generated hashes have been validated against Cyclone DDS.

**type_identifier** (`type_identifier: [32]u8`): SHA-256 fingerprint.
This is a zidl-internal convention, not transmitted. Useful for out-of-band
tooling (type registries, schema databases) that needs a stronger identifier
than the 14-byte EquivalenceHash.

**What's deferred:**
- TypeObject for `union`, `bitmask`, `bitset` (currently emits TK_NONE placeholder).
- `typedef`/alias TypeObjects.
- MUTABLE type support in TypeObject (and in CDR serialization).

## `--generate-interfaces`: DCPS API Generation

When `--generate-interfaces` is set, each IDL `interface` declaration becomes
a fat-pointer vtable struct in Zig, an abstract class in C++, an interface in
Java, or a vtable struct in C.

Additionally, for each IDL `struct` that is a DDS topic type (not `@nested`),
the generated code should include the typed DataWriter/DataReader wrappers.
The full normative interface IDL is in `docs/dcps_idl.md`.

**How topic types are determined:**
- A struct with `@topic` annotation: `TypeAnnotations.is_topic == true`.
- A struct without `@nested`: `TypeAnnotations.is_nested == false` (default).
- Structs with `@nested` are helper types embedded inside other types —
  they do not get DataWriter/DataReader.

**Generated interface shape** (Zig example for a struct `Foo`):

```zig
// FooTypeSupport — wraps registration
pub const FooTypeSupport = struct { ... };

// FooDataWriter — typed write operations
pub const FooDataWriter = struct {
    ptr: *anyopaque,
    vtable: *const Vtable,
    pub const Vtable = struct {
        register_instance: *const fn (*anyopaque, *const Foo) i32,
        write: *const fn (*anyopaque, *const Foo, i32) i32,
        dispose: *const fn (*anyopaque, *const Foo, i32) i32,
        // ...
    };
};

// FooDataReader — typed read operations
pub const FooDataReader = struct { ... };
```

## RTPS Transport (Future)

The forthcoming Zig DDS runtime will implement RTPS 2.3 (formal/22-04-01):

- Participant discovery (SPDP / SEDP).
- UDP/IP transport (and eventually shared-memory).
- RTPS message framing (Header, SubMessage, Info_TS, etc.).
- Reliable and best-effort delivery.

zidl's CDR encoding is aligned with RTPS §9.6 (XCDR2 data submessages).
The `encap header` format emitted by `CdrWriter.writeEncapHeader()` matches
the RTPS SerializedPayload format (4-byte header with representation identifier).

---

## DDS-RPC (Future, Design-Preserved)

DDS-RPC v1.0 (formal/17-04-01) enables request-reply patterns over DDS topics.
The zidl architecture must not preclude DDS-RPC support:

- IDL `interface` operations map to request/reply topic pairs in DDS-RPC.
- The `--generate-interfaces` flag is the hook for this; the current Zig vtable
  output is the local-call form. A future `--rpc` flag could emit request/reply
  struct types and serialization instead.
- **Decision**: do not conflate `--generate-interfaces` with DDS-RPC — keep
  them orthogonal. `--generate-interfaces` is for the DCPS abstract API;
  DDS-RPC is a separate code-gen path.

---

## DDS Security (Future)

DDS Security v1.2 (formal/25-03-06) adds authentication, access control,
and cryptographic protection layers. This is outside current scope but should
not be precluded by architecture decisions.

---

## Encoding Beyond CDR

JSON, XML, TOML, and other non-CDR encodings are **not** a code-generation concern. The
right approach is a schema-driven runtime that uses TypeObject as the schema descriptor:

- Every generated type already carries pre-computed `type_object` bytes (Zig backend today).
- A DynamicType/DynamicData implementation (owned by the forthcoming DDS implementation)
  parses those bytes and drives encoding-agnostic serialization.
- XCDR↔JSON, XCDR↔XML, MQTT bridge, config-file serialization etc. are all DDS runtime
  features, not backend features. No `--encoding` flag on zidl's CLI is needed.

DynamicType/DynamicData and the TypeObjectFactory belong in the DDS implementation, not in
zidl or `zidl-xtypes`. The `zidl-xtypes` package provides only the constants (EK_*/TK_*/TI_*/FLAG_*)
that both zidl and the DDS runtime need to share.

## Compliance Profiles (from DDS v1.4 §A)

The DDS v1.4 spec defines five compliance profiles:

| Profile | Optional features added |
|---|---|
| Minimum | Mandatory DCPS only |
| Content-subscription | ContentFilteredTopic, QueryCondition, MultiTopic |
| Persistence | DURABILITY_SERVICE QoS, TRANSIENT + PERSISTENT durability kinds |
| Ownership | EXCLUSIVE_OWNERSHIP, OWNERSHIP_STRENGTH, HISTORY depth > 1 |
| Group access | PRESENTATION access_scope = GROUP |

zidl's `--generate-interfaces` targets the full DCPS API (all five profiles).
A future `--profile dcps-min` flag could suppress optional interfaces.
